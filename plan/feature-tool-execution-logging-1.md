# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan defines exact implementation steps to ensure every MCP tool execution is logged consistently with start, success, and error events, including tool identity and key request context.

## 1. Requirements & Constraints

- **REQ-001**: Log every tool invocation for all tools exposed by the server, including db_01_* and db_02_* names.
- **REQ-002**: Emit three standardized events per execution path: start, success, and error.
- **REQ-003**: Include tool name, instance, database, schema, table, and request id/session id when available.
- **REQ-004**: Avoid logging sensitive fields such as raw passwords and full prompt text.
- **SEC-001**: Do not include DB credentials in any log line.
- **SEC-002**: Respect existing audit log settings and avoid duplicating SQL text in server logs when not required.
- **OBS-001**: Preserve compatibility with current `MCP_LOG_LEVEL` and file rotation behavior.
- **CON-001**: Keep existing tool return payloads unchanged.
- **CON-002**: Keep existing tool signatures unchanged to avoid MCP schema drift.
- **GUD-001**: Use centralized logging helpers to prevent drift across tools.
- **PAT-001**: Prefer wrapper-based instrumentation at MCP registration boundary to guarantee full coverage.

## 2. Implementation Steps

- GOAL-001: Introduce a centralized execution logging contract and helpers in the server runtime.

| TASK-002 | Add helper function `_log_tool_start(tool_name: str, args: dict[str, Any]) -> None` in mcp_sqlserver/server.py to emit `tool.start` log records via `logger.info`. |  |  |
| TASK-003 | Add helper function `_log_tool_success(tool_name: str, elapsed_ms: int, result_meta: dict[str, Any]) -> None` in mcp_sqlserver/server.py to emit `tool.success`. |  |  |
| TASK-004 | Add helper function `_log_tool_error(tool_name: str, elapsed_ms: int, exc: Exception) -> None` in mcp_sqlserver/server.py to emit `tool.error` with `logger.exception`. |  |  |
| TASK-005 | Add helper function `_extract_result_meta(result: Any) -> dict[str, Any]` in mcp_sqlserver/server.py that reports safe summary fields such as `row_count`, `keys_present`, and `status`. |  |  |

## 3. Alternatives

- **ALT-001**: Add manual `logger.info` statements inside each tool function. Rejected because it is error-prone and likely to miss future tools.
- **ALT-002**: Log only at transport layer (FastMCP request processing). Rejected because it does not provide stable per-tool context fields.
- **ALT-003**: Rely only on audit query logs. Rejected because non-query tools and failed pre-query validations would remain invisible.

## 4. Dependencies

- **DEP-001**: Python standard library `logging` and `time` modules.
- **DEP-002**: Existing dynamic tool registration implementation in mcp_sqlserver/server.py.
- **DEP-003**: Existing runtime log configuration controlled by `MCP_LOG_LEVEL`, `MCP_LOG_FILE`, and rotation keys.

## 5. Files

- **FILE-001**: mcp_sqlserver/server.py - add centralized logging helpers and wrapper instrumentation.
- **FILE-002**: tests/test_integration_tools.py - add log visibility and error-path assertions.
- **FILE-003**: .env.example - optional feature flag documentation for execution logging enablement.
- **FILE-004**: README.md - document log event format and sample entries.

## 6. Testing

- **TEST-001**: Compile check: `python -m py_compile mcp_sqlserver/server.py`.
- **TEST-002**: Integration tests: `pytest -v tests/test_integration_tools.py`.
- **TEST-003**: Runtime verification: call `db_01_table_health(instance=1,database_name=USGISPRO_800,schema=dbo,table_name=Account)` and assert `tool.start`/`tool.success` lines in `/app/logs/server.log`.
- **TEST-004**: Failure verification: call tool with missing required fields and assert `tool.error` line in `/app/logs/server.log`.
- **TEST-005**: Privacy verification: confirm sensitive keys are absent from emitted log JSON payload.

## 7. Risks & Assumptions

- **RISK-001**: High-volume logging at DEBUG/INFO may increase log size and I/O overhead.
- **RISK-002**: Incomplete sanitization could expose sensitive metadata if new parameter names are introduced later.
- **RISK-003**: Wrapper refactor at registration layer could impact tooling if callable signatures are altered.
- **ASSUMPTION-001**: All externally exposed tools are registered through the shared dynamic registration path.
- **ASSUMPTION-002**: Container runtime has writable path for `/app/logs/server.log`.
- **ASSUMPTION-003**: Existing `MCP_LOG_LEVEL` is set to at least INFO in environments where visibility is required.
